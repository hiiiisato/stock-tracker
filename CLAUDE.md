# 株式分析アプリ 開発指針

このファイルはClaude Codeがこのプロジェクトで作業する際の指示書です。

## UI / デザイン

- **スマホ対応を常に意識すること。** PCとスマホの両方で動作確認を想定し、レスポンシブデザインを徹底する。メディアクエリ（`max-width: 768px` など）を必ず設け、モバイルで表示が崩れないよう設計する。
- モダンなダークテーマ（GitHub風）を維持する。新しいUIコンポーネントを追加する際も既存のカラーパレット・フォントサイズ・角丸・ボーダー色に揃えること。

## インフラ・自動化

- **すべての定期処理はサーバー側で完結させること。** Mac のスリープ・電源 OFF に関係なく動作することが前提。
  - 日次バッチ（`daily_run.py`）は Render.com の cron ジョブ（`render.yaml` の `daily-stock-update`）で自動実行される。平日 17:00 JST（UTC 8:00）に動作。
  - 新しい定期処理を追加する場合も、Mac での手動実行ではなく Render.com cron / GitHub Actions など **サーバー側のスケジューラ**に登録すること。
  - ローカル実行はデバッグ・臨時実行のみ。本番の定期処理をローカル cron に依存させない。

## データ管理

- **データが変わるものは、更新・メンテナンスの仕組みまでセットで設計すること。**
  - 株価・出来高: `daily_run.py` による日次バッチ（Render.com cron で自動実行）
  - 業績・財務指標: `financials.py` / `financials_kabutan.py` による定期取得
  - 株価変動要因: `event_researcher.py` → `daily_run.py` 経由で自動実行（Gemini AI 要約付き）
  - 銘柄マスタ: 新規上場・上場廃止の追加・無効化フローを考慮すること
  - 新しいデータソースを追加する場合も、同様に「どう最新化するか」を実装まで含めて提案・実装する。
- DBスキーマを変更する場合は、既存データとの互換性（`ALTER TABLE` or `DEFAULT` 付き `ADD COLUMN`）を考慮し、初回マイグレーション手順をコメントに残す。

## コスト方針

- **原則として無料で実現する。** 外部API・サービスは無料枠の範囲で設計すること。
- 有料サービスが選択肢になる場合は、「費用感・得られるメリット・無料代替案との差分」をセットで提案すること。ユーザーが判断できる情報を添えた上で提案し、勝手に有料サービスを組み込まない。

## 拡張性

- **機能追加が続くことを前提に設計する。** 以下を意識すること：
  - DBテーブルは後からカラム追加しやすいよう、`NULL` 許容・デフォルト値を活用する
  - 新しいページ・セクションを追加しやすいよう、ルーティング・共通UI部品（`_nav`, `_page_html` 等）の構造を壊さない
  - ハードコードを避け、設定値は `config.py` やDBマスタで管理する
  - データソースが増えても対応できるよう、取得ロジックとDB保存ロジックを分離しておく

## 開発スタイル

- **モダンな書き方を徹底する。** Python 3.10+ の型ヒント・f-string・辞書内包表記などを積極活用する。
- セキュリティ: SQLはすべてプレースホルダー（`%s`）を使用。ユーザー入力をHTMLに直接埋め込む場合は必ずエスケープする。
- パフォーマンス: 重いページ生成にはインメモリキャッシュ（`_get` / `_set`）を活用し、不要なN+1クエリを避ける。
- 新機能を追加する際は、データがない状態（0件・NULL）でもエラーにならないフォールバック処理を入れる。
